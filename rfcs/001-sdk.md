# RFC 001: perfscale SDK

- **Status**: Draft
- **Author**: Perfscale Team
- **Created**: 2026-07-09
- **Requires**: none
- **Required by**: RFC 002 (Marketplace)

## Summary

Define an official SDK surface for perfscale: a stable, documented way to
define, run, and observe load tests from code — without shelling out to the
`perfscale` binary and scraping its stdout. The SDK consists of three layers
with very different costs: (1) a stabilized `perfscale-core` Rust API, (2) a
portable **test-definition contract** (the YAML/JSON schema) as the true
cross-language interface, and (3) thin language SDKs (JS first, Python
second) that build definitions and drive execution through the agent/CLI
rather than through FFI.

## Motivation

Today there are exactly three ways to use perfscale programmatically, and
all of them are accidental interfaces:

1. **Depend on `perfscale-core` from git** — this is what `perfscaled` does
   (`git = "...", branch = "main"`). It works, but every internal refactor is
   a breaking change; the `run_steps` signature changed twice in one week
   (`quiet` parameter, histogram metrics) and broke the agent build both
   times. There is no semver, no stability promise, no documented surface.
2. **Shell out to the CLI and parse stdout** — the GitHub Action does this.
   `--summary-export` made it tolerable, but test *construction* is still
   string-templating YAML.
3. **POST step JSON to a perfscaled agent** — the controlplane does this.
   The wire shape (`StepsRunRequest`) is a de-facto API with no versioning.

An `sdk-js` repository already exists in the org with ambitions but no
engine contract to build on. Meanwhile RFC 002 (Marketplace) cannot define
"what is an action" without this RFC defining who owns that vocabulary.

## Goals

- A named, versioned, documented **public API** with semver guarantees.
- Test definitions constructible in TypeScript/Python with type safety and
  editor autocomplete, runnable locally (CLI) or on a fleet (agent API).
- Programmatic access to results (`RunSummary` / `SummaryExport` shapes).
- `perfscale-core` consumable as a Rust library by third parties (the
  embedded engine case: run load tests inside an existing Rust service).

## Non-goals

- **In-process bindings (FFI/napi/pyo3) for the engine.** See Pitfalls — the
  cost/benefit is wrong at the current stage.
- Browser execution (WASM). The engine's value is sockets at scale; WASM has
  neither raw sockets nor the scheduler control we need.
- Replacing k6/locust scripting. The SDK builds *native* test definitions;
  k6 scripts remain k6's domain.

## Detailed design

### Layer 1 — the contract: test definition schema as the API

The single most important decision: **the YAML/JSON test definition and the
summary export are the SDK's wire contract**, not any Rust type or function
signature. They already have generated JSON Schemas (`schema/`), a validator
(`perfscale lint`), an idempotent execution endpoint (`POST
/api/v1/run/steps`), and a machine-readable result (`SummaryExport`).

Actions to take:

- Version the schema explicitly (`schemaVersion: 1` field, additive-only
  within a major).
- Publish schemas to a stable URL (site already serves docs; schemas join
  them) so non-Rust tooling can validate offline.
- Document the **compatibility policy**: unknown fields are ignored at
  runtime (already true — lint warns instead), so older engines tolerate
  newer definitions degrading gracefully.

### Layer 2 — Rust: stabilize a `perfscale` facade crate

`perfscale-core` today exports everything (`pub mod` all the way down),
including internals like `Metrics` (made `pub` only for benches, marked
`#[doc(hidden)]`). Freezing all of it would fossilize the engine.

Instead: a new thin crate `perfscale` (the good crates.io name, currently
unused) re-exporting a curated surface:

```rust
pub use perfscale_core::step::{TestDef, Step, RunConfig};
pub use perfscale_core::yaml::{parse_test_file, parse_config_file};
pub use perfscale_core::summary::{RunSummary, SummaryExport, parse_summary};
pub struct Runner { /* owns tokio handle wiring, quiet, channels */ }
impl Runner {
    pub async fn run(&self, test: TestDef, config: RunConfig) -> RunReport;
    pub fn stream(&self, ...) -> impl Stream<Item = LogLine>;
}
```

`perfscale-core` stays internal-moving (0.x forever if needed); the facade
gets real semver. perfscaled migrates from git-main to the facade, which
retroactively fixes the "agent breaks on every engine refactor" problem.

### Layer 3 — language SDKs: builders + drivers, not engines

`@perfscale/sdk` (TypeScript, the existing `sdk-js` repo) and later
`perfscale` (PyPI):

- **Builder**: typed construction of test definitions.
  `test().step(http.get(url).check({ status: 200 }).outputs("resp"))` →
  emits schema-valid JSON/YAML. Types generated from the JSON Schema
  (single source of truth; `typeshare` is already used in controlplane for
  the same trick).
- **Driver**: three interchangeable executors behind one interface:
  local CLI subprocess (`--summary-export` JSON as the result channel),
  perfscaled agent HTTP (`StepsRunRequest`, SSE for live lines), and —
  later — controlplane task API for fleet runs.
- **Results**: parse `SummaryExport`; expose percentiles/RPS/error-rate as
  typed objects; helpers for CI gating (`assertP95Below(500)`).

No engine code crosses the language boundary. The SDKs are ~thin and
therefore cheap to maintain — the engine's correctness burden stays in one
place.

## Benefits

- **Stops the accidental-API bleed**: three existing integrations
  (perfscaled, GitHub Action, controlplane) move onto surfaces with names
  and versions; internal refactors stop breaking downstreams.
- **Programmable tests unlock real use cases**: parameterized scenario
  matrices, test generation from OpenAPI specs, CI gates as code instead of
  jq one-liners.
- **Foundation for RFC 002**: a marketplace needs a stable definition of
  "action", "test definition", and "result" — this RFC provides all three.
- **Adoption funnel**: `npm install @perfscale/sdk` is a smaller first step
  than "install a Rust binary and learn our YAML".

## Drawbacks

- **Semver is a treadmill.** Once `perfscale` 1.0 ships, every engine
  improvement must be triaged for breakage. The facade contains this but
  doesn't eliminate it: someone must own deprecation cycles.
- **Three repos to keep in sync** (engine, sdk-js, future sdk-py) with a
  two-person team. Schema-generated types reduce but don't remove drift.
- **Documentation surface doubles**: every feature now needs YAML docs *and*
  SDK docs (the `${{ ... }}`/multipart/file-actions docs just written would
  each need SDK equivalents).

## Tradeoffs

- **Facade crate vs. stabilizing perfscale-core directly**: the facade costs
  one more crate and re-export boilerplate, but buys freedom to keep
  refactoring internals (histogram change, quiet parameter would have been
  invisible behind `Runner`). Chosen: facade.
- **Subprocess/HTTP drivers vs. native bindings**: bindings give in-process
  speed and no binary management, but load generation is not
  latency-sensitive on the *control* path — a subprocess launch per run is
  noise against a 30-second test. Chosen: subprocess/HTTP.
- **Generated types vs. handwritten SDK types**: generated types are always
  correct but produce clumsy ergonomics; handwritten builders are pleasant
  but drift. Chosen: generate the *data* types, handwrite the *builder*
  sugar on top.

## Non-obvious pitfalls

1. **Global state in the engine.** `shared_client()`, the insecure client,
   and the `std/file-read` cache are process-wide `OnceLock`s. Fine in a
   CLI; surprising in an embedded library (two `Runner`s in one process
   share connection pools and file caches; a test suite embedding the
   engine can leak state between tests). The facade must either document
   this loudly or scope these per-`Runner` before 1.0.
2. **Tokio runtime ownership.** `run_steps` assumes it's inside a runtime.
   Embedders with their own runtime (or none) hit panics. The facade needs
   both `async fn run` and a blocking `run_blocking` that owns a runtime.
3. **The stdout contract is sneaky-load-bearing.** The k6-compatible summary
   text is parsed by perfscaled, the controlplane, and bench scripts. SDK
   work will tempt someone to "clean up" the text format — that text *is*
   an API with three consumers. Formalize it or freeze it.
4. **Schema "additive-only" is weaker than it sounds.** Adding a field is
   non-breaking for parsers but can be semantically breaking for executors:
   an old engine silently *ignoring* `multipart:` would send an empty-body
   POST — a wrong load test that looks green. The engine needs a
   `schemaVersion` gate that *rejects* definitions newer than it
   understands, which is the opposite of lenient parsing. This subtlety is
   where most "compatible" schema evolutions actually break users.
5. **Windows.** The CLI builds for Windows; the subprocess driver must
   handle `.exe` resolution, path quoting, and the lack of SIGTERM. The
   GitHub Action already fights this — SDK inherits it.
6. **Version skew testing is a matrix, not a point.** SDK 1.2 × engine
   0.3/0.4/0.5 × agent API vN — CI must run a compat matrix or the promise
   is fiction. Budget it: it's the single biggest hidden cost of this RFC.
7. **crates.io name.** `perfscale` must be claimed before this RFC is
   public; name squatting on crates.io is unfixable after the fact.

## Alternatives considered

- **Do nothing** (keep git-main dependency + stdout scraping): the current
  breakage rate already costs more than the facade would.
- **HTTP-only SDK** (agent API as the sole driver): cleanest contract, but
  forces every local dev loop through a daemon; kills the "quick local run"
  ergonomics that make the CLI pleasant.
- **FFI bindings now**: maximal power, maximal maintenance; napi/pyo3
  multiply the build matrix by platforms × Node/Python versions. Revisit
  when the engine's API has survived a year of facade-level stability.

## Rollout plan

1. Claim `perfscale` on crates.io, `@perfscale/sdk` on npm (exists), and
   `perfscale-sdk` on PyPI. (One afternoon; irreversible if skipped.)
2. Facade crate with `Runner` + curated re-exports; migrate perfscaled to
   it; delete `#[doc(hidden)]` leaks from the public story.
3. `schemaVersion` field + rejection gate + published schema URLs.
4. TypeScript SDK: generated types → builder → CLI driver → agent driver.
5. Python SDK after TS proves the shape.

## Open questions

- Does the facade expose the k6/locust runners, or is the SDK
  native-engine-only? (Leaning: native-only; k6 users already have k6.)
- Where do SDK docs live — the OSS repo's `docs/` (and thus `/docs/oss` on
  the site) or per-SDK repos? Split docs rot faster.
- Is `RunReport` streaming-first (Stream of events with a final summary) or
  summary-first with an optional stream? CI wants the latter, dashboards
  the former.

## Success metrics

- perfscaled builds against a tagged facade version, not git-main.
- One external project embeds the facade or the TS SDK within two release
  cycles.
- Zero "engine refactor broke the agent" incidents in the quarter after
  migration.
